# Proposta: Acoes Contextuais no Checklist por Reboque

<StatusBadge status="em-discussao" />

**Data:** 2026-07-10

---

## Problema

Ao marcar um item como **Nao Conforme** no checklist, o usuario pode querer criar um servico ou solicitar uma peca. Hoje os botoes de acao aparecem por **row** (item template), mas uma row pode ter **multiplos reboques** — nao ha como saber a qual reboque a acao se refere.

### Fluxo atual

```
[ Verificar Pneus ]  [✗ ✓ …]SR1  [✗ ✓ …]SR2  [✗ ✓ …]SR3   [📦 🔧]
                                                                  ↑
                                                          Qual reboque?
```

- Os botoes 📦 (Solicitar Peca) e 🔧 (Adicionar Servico) aparecem como overlay na row
- O dialog de "Adicionar Servico" tem um select de Reboque, mas abre sem pre-selecao
- O usuario precisa lembrar qual reboque estava nao conforme e selecionar manualmente

---

## Opcoes

### Opcao 1 — Estender o dropdown existente (tres pontos)

Cada celula de conformidade ja possui um dropdown (tres pontos) por reboque. Quando o status e diferente de `PENDING`, adicionar itens de acao ao menu:

```
┌──────────────────────┐
│ ⊘  Nao se Aplica     │
│ ⚠  Parcial           │
│ ○  Limpar status     │
│──────────────────────│  ← separador (so aparece se status ≠ PENDING)
│ 🔧 Adicionar Servico │
│ 📦 Solicitar Peca    │
└──────────────────────┘
```

**Comportamento:**
- "Adicionar Servico" abre o dialog com o trailer pre-selecionado no form
- "Solicitar Peca" navega para a tela de criacao (pode passar trailerId como query param)
- As acoes extras so aparecem quando status ≠ PENDING (item ja foi avaliado)

**Pros:**
- Zero mudanca visual — usa trigger que ja existe em cada celula
- Contexto claro — o dropdown pertence a um trailer especifico
- Implementacao simples — so adicionar items ao DropdownMenu e passar trailerId ao openDialog

**Contras:**
- Menos descobrivel — acoes escondidas dentro do menu de tres pontos
- Mistura mudanca de status com acoes de negocio no mesmo menu

---

### Opcao 2 — Icone de acao na celula nao conforme

Quando uma celula e marcada como `NON_CONFORM`, um icone de acao (ex: `+` ou `🔧`) aparece abaixo ou ao lado dos botoes de conformidade naquela celula.

```
  SR1          SR2          SR3
[✗  ✓  …]   [✗  ✓  …]   [✗  ✓  …]
   🔧                        🔧      ← so nas celulas NON_CONFORM
```

Clicar abre um mini-popover com as opcoes:
```
┌──────────────────────┐
│ 🔧 Adicionar Servico │
│ 📦 Solicitar Peca    │
└──────────────────────┘
```

**Pros:**
- Altamente visivel e descobrivel
- Contexto explicito — esta dentro da celula do trailer
- Feedback visual claro de que ha acoes disponiveis

**Contras:**
- Adiciona ruido visual quando muitas celulas sao non-conform
- Pode precisar de ajuste na largura da coluna (120px) dependendo do tamanho do icone

---

### Opcao 3 — Expandir row com detalhe por reboque

Clicar na row (ou em um chevron) expande uma sub-secao mostrando os reboques nao conformes com botoes de acao dedicados.

```
[ Verificar Pneus ]  [✗ ✓ …]  [✗ ✓ …]  [✗ ✓ …]    ▼
├── SR1 — Nao Conforme  [🔧 Adicionar Servico] [📦 Solicitar Peca]
└── SR3 — Nao Conforme  [🔧 Adicionar Servico] [📦 Solicitar Peca]
```

**Pros:**
- Visao clara de todos os reboques com problema
- Espaco para mostrar informacoes adicionais (observacoes, fotos)
- Bom para fluxos futuros mais complexos (ex: adicionar nota por reboque)

**Contras:**
- Muda o layout da lista (scroll, altura variavel)
- Implementacao mais complexa
- Pode ficar pesado se muitas rows estiverem expandidas

## Mudancas tecnicas necessarias (comum a todas as opcoes)

### 1. Store do dialog precisa aceitar trailerId

```typescript
// use-create-service-execution-dialog.ts
interface AddServiceExecutionDialogState {
  isOpen: boolean;
  assignmentId: string | null;
  preSelectedTrailerId: string | null;  // novo
  openDialog: (trailerId?: string) => void;
  closeDialog: () => void;
}
```

### 2. Form pre-preenche trailer quando recebido

```typescript
// add-service-execution.tsx
useEffect(() => {
  if (preSelectedTrailerId && assetType === AssetTypeEnum.TRAILER) {
    form.setValue("trailerId", preSelectedTrailerId);
  }
}, [preSelectedTrailerId]);
```

### 3. Rota de "Solicitar Peca" aceita trailerId (opcional)

```
/part-request/create/:workOrderId?trailerId=xxx
```

## Recomendacao

**Opcao 1** e a mais pragmatica — zero mudanca visual, implementacao leve, e resolve o problema de contexto. O dropdown ja e o "ponto de acao" natural por celula.

Se a descobribilidade for uma preocupacao, **Opcao 2** e uma boa alternativa, mas precisa de cuidado com o design para nao poluir a grid.

**Opcao 3** faz sentido como evolucao futura se o checklist ganhar mais features por celula (observacoes, fotos, historico).
